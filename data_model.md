# Data Model Specification

## 1. Overview

This document defines the data structure used in the Sensor to Web IoT system.

The system stores sensor readings in MongoDB as time-series data.

Each reading represents a snapshot of sensor values collected from the ESP32 device.

---

## 2. Database Configuration

Database: MongoDB
Database Name: sensor_to_web_iot
Collection Name: readings

---

## 3. Document Structure

Each document in the collection represents one sensor reading.

### Example Document

{
"_id": "ObjectId",
"sensor_id": "esp32_01",
"temperature": 25.4,
"humidity": 60.2,
"light": 0,
"obstacle": 1,
"potentiometer": 1200,
"timestamp": "2026-04-15T20:38:38Z"
}

---

## 4. Field Definitions

### _id

* Type: ObjectId
* Description: Unique identifier automatically generated by MongoDB

---

### sensor_id

* Type: string
* Description: Identifier of the sensor device
* Example: "esp32_01"

---

### temperature

* Type: float
* Description: Temperature value in Celsius
* Source: simulated

---

### humidity

* Type: float
* Description: Humidity value in percentage
* Source: simulated

---

### light

* Type: integer (0 or 1)
* Description:

  * 1 = dark
  * 0 = light detected

---

### obstacle

* Type: integer (0 or 1)
* Description:

  * 0 = obstacle detected
  * 1 = clear

---

### potentiometer

* Type: integer (0–4095)
* Description: Analog sensor value

---

### timestamp

* Type: datetime (UTC)
* Description: Time when the reading was stored
* Generated by backend

---

## 5. Data Lifecycle

1. ESP32 generates sensor data
2. Data is sent via HTTP POST
3. Backend validates and stores data
4. MongoDB assigns _id
5. Timestamp is added at insertion
6. Data is queried by dashboard

---

## 6. Data Access Patterns

### Latest Data

* Query: sort by timestamp descending
* Limit: 1

### Recent Data

* Query: sort by timestamp descending
* Limit: configurable (default 20)

---

## 7. Indexing Strategy

Recommended index:

timestamp (descending)

Purpose:

* Improve query performance for latest and recent data

Example:

db.readings.createIndex({ timestamp: -1 })

---

## 8. Data Integrity

Validation is handled at backend level using Pydantic.

Rules:

* All required fields must exist
* Data types must match schema
* Invalid requests are rejected

---

## 9. Scalability Considerations

Future improvements:

* Partition data by sensor_id
* Use time-series collections (MongoDB feature)
* Archive old data
* Use dedicated time-series database (InfluxDB)

---

## 10. Limitations

* Single device (sensor_id not heavily used yet)
* No data compression
* No retention policy
* Simulated temperature and humidity

---

## 11. Future Enhancements

* Support multiple devices
* Add metadata (location, device type)
* Add derived metrics (average, trends)
* Implement data expiration policy (TTL index)
